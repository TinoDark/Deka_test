## 3. Service: Railway + Docker

### Dockerfile Backend Production
```dockerfile
FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production

COPY dist ./dist

EXPOSE 3000

CMD ["node", "dist/main.js"]
```

### Dockerfile Frontend Production
```dockerfile
FROM node:18-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

FROM node:18-alpine

WORKDIR /app
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./package.json

EXPOSE 3000

CMD ["npm", "start"]
```

### Railway Deploy
```bash
# 1. Login Railway
railway login

# 2. Link à projet
railway link

# 3. Ajouter PostgreSQL plugin
railway add

# 4. Configurer variables
railway env

# 5. Déployer
railway up
```

## 4. Variables Critiques par Environnement

### 🔐 Secrets Sensibles

| Variable | Local | Production | Source |
|----------|-------|------------|--------|
| `JWT_SECRET` | `deka-secret-key-change` | **UNIQUE 32+ chars** | Générer |
| `DATABASE_URL` | `localhost:5432` | **Railway/AWS RDS** | Railway console |
| `GOOGLE_MAPS_API_KEY` | Test key | **Production key** | Google Cloud |
| `PAYGATEGLOBAL_API_KEY` | `5c08692e...` | **Clé production** | PayGateGlobal |
| `ADMIN_PASSWORD` | `admin123` | **Strong password** | Générer |

### 🟢 Variables Non-Sensibles (Public)

```
NEXT_PUBLIC_API_URL
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY (restriction domaine)
NODE_ENV
API_PORT
```

## 5. Générer JWT_SECRET Sécurisé

```bash
# Méthode 1: Node.js
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# Méthode 2: OpenSSL
openssl rand -hex 32

# Méthode 3: bash
head -c 32 /dev/urandom | base64

# Output: 7f3e9c2a1b4d8f6e5a9c2b1d4e7f8a9c
```

Retournez cette clé dans Railway console → Variables.

### Workflow Deployment

```yaml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Build Backend
        run: |
          cd backend
          npm ci
          npm run build
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
      
      - name: Deploy to Railway
        run: railway up
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
```

## 11. Gestion Changements API Keys

### Rotation Keys (Tous les 90 jours)

```bash
# 1. Générer nouvelle clé
new_key=$(node -e "console.log(require('crypto').randomBytes(32).toString('hex'))")

# 2. Tester en staging
railway env --env staging
# Remplacer JWT_SECRET = $new_key

# 3. Valider 24h
# Tests complets...

# 4. Déployer en production
railway env --env production
# Remplacer JWT_SECRET = $new_key

# 5. Monitorer logs
railway logs --service backend
```

### Révoquer Keys Compromises

```bash
# 1. Si fuites:
new_key=$(openssl rand -hex 32)

# 2. Update immédiatement
railway env --env production
# JWT_SECRET = $new_key

# 3. Restart services
railway restart

# 4. Invalider tokens existants
# Ajouter dans DB: token_blacklist
```

## 13. Récapitulatif: Meilleure Pratique Production

✅ **À Faire:**
- Utiliser Railway/Vercel console pour secrets
- Générer JWT_SECRET fort (32+ chars random)
- Restreindre Google Maps par domaine
- Utiliser HTTPS partout
- Monitorer accès logs
- Rotation keys tous 90 jours
- Chiffrement data sensitive

❌ **À NE PAS Faire:**
- Coder secrets dans code source
- Commiter fichiers .env en production
- Utiliser même clé dev/prod
- Diffuser passwords par email
- Laisser secrets en plain text
- Oublier HTTPS en production
